# 七年级英语听说课多模态生成与质量评测工作台 — 产品需求摘要

> 基于 PRD V0.1 整理 | 日期：2026-07 | 首版主题：Asking for and Giving Directions

---

## 一、产品背景和目标

### 业务背景

初中英语听说课需要同时准备对话脚本、情境图片、听力音频、练习题和答案。教师当前做法是人工编写或分别使用多个AI工具生成不同模态内容，再手工拼合到课件中。关键问题：
1. 不同模态之间缺少统一的数据关系和版本绑定
2. 缺少跨模态一致性的质量检查
3. 错误不可定位，教师不知从何修改
4. 修复后缺少回归机制

### 竞品启示

- **MagicSchool**：结构化表单 + Prompt助手 + 报告化反馈，但多模态输出分散在不同工具中
- **Curipod**：课程生成 + 课堂互动闭环，但实测不支持直接生成音频，且无Bad Case修复闭环

### 产品目标

| 层级 | 目标 |
|------|------|
| 用户目标 | 一套工作台完成听说课素材的生成 → 检查 → 修复 → 导出 |
| 质量目标 | 降低S3/S4严重错误（语法、答案、跨模态冲突）进入课堂的概率 |
| 效率目标 | 支持局部修复而非整体重做，减少人工核对成本 |
| 产品目标 | 形成可演示、可评测、可迭代的AI产品闭环 |

---

## 二、目标用户

**核心用户**：七年级英语一线教师

- 具备基本办公软件能力，不要求熟悉Prompt工程或TTS参数
- 核心关注：生成结果是否符合教材进度 → 能否直接修改 → 能否减少人工核对 → 最终能否放心用于课堂
- 使用频率：以单元备课/公开课需求触发，不一定每天使用

**扩展用户**：备课组/教研员、教育内容制作人员

**非目标用户（V0.1不支持）**：学生、学校管理者、其他年级/学科教师、高风险自动决策场景

## 三、核心业务流程

```
配置（新建任务）→ 脚本生成 → 脚本审核（文本门禁）→ 确认脚本版本
→ 多模态生成（图片+音频+题目）→ 评测报告 → Bad Case修复 → 终审导出
```

### 系统执行节点

| 节点类型 | 职责 |
|----------|------|
| 规则节点 | 输入校验、数量检查、版本检查、文件状态检查 |
| LLM节点 | 脚本生成、题目生成、语义质量评测 |
| 图像节点 | 情境图生成与图像评测 |
| 语音节点 | TTS合成、ASR回转比对 |
| 判断节点 | 文本门禁、总质量门禁 |
| 人工节点 | 脚本确认、最终审核、评测纠错 |

## 五、核心功能（P0，12项）

| 编号 | 功能 | 说明 |
|------|------|------|
| F01 | 任务配置 | 填写主题、词汇、句型、轮次、时长、题数 |
| F02 | 输入校验 | 缺失/越界/冲突时阻止提交并定位字段 |
| F03 | 脚本生成 | 按七年级难度生成双角色结构化对话 |
| F04 | 脚本审核 | 编辑+重生成+文本门禁+确认锁定版本 |
| F05 | 图片生成 | 根据脚本实体和空间关系生成情境图 |
| F06 | 音频生成 | 根据确认脚本+音色/语速生成双角色音频 |
| F07 | 题目生成 | 生成N道单选听力题+选项+唯一答案 |
| F08 | 单模态评测 | 分别检查文本/图片/音频/题目/答案质量 |
| F09 | 跨模态一致性评测 | 检查图文/文音/题音/答案/版本一致性 |
| F10 | Bad Case诊断 | 展示错误位置+证据+严重程度+根因+修复建议 |
| F11 | 局部修复 | 只重做出错模块，不清空其他模块 |
| F12 | 审核导出 | 门禁+教师审核后打包导出 |

### P1功能（7项）

上传已有脚本(P1-01)、上传素材质检(P1-02)、任务模板(P1-03)、多模型对比(P1-04)、批量任务(P1-05)、自定义Rubric(P1-06)、团队共享(P1-07)

## 六、数据对象

### 四大数据实体

| 实体 | 核心字段 | 说明 |
|------|----------|------|
| 任务配置 (Task) | task_id, topic, grade, scenario, required_vocabulary, target_patterns, dialogue_turns, audio_duration_target_sec, question_count | 生成约束输入 |
| 生成结果 (Generation) | script_id/version, image/audio/question IDs + source_script_version + model_name/prompt_version | 每个模态独立版本 |
| 评测记录 (Evaluation) | evaluation_id, target_type/id, overall_score, severity(S0-S4), error_type, evidence, suspected_cause, repair_suggestion, teacher_feedback | 可追溯到具体素材版本 |
| 工作流日志 (Workflow) | workflow_run_id, node_name, status, input/output_version_refs, retry_count | 审计和恢复 |

### 版本依赖规则

- 图片/音频/题目的 `source_script_version` 必须等于当前 `script_version`
- 脚本修改 → 旧下游素材自动标记为 `outdated`
- 评测记录绑定具体素材版本，不模糊引用"最新结果"
- 版本冲突时阻止导出

## 七、状态流转

```
draft → generating → partial_success → evaluating → (needs_fix →) pending_review → approved → exported
                    ↘ failed（可重试）
                                         ↘ outdated（脚本过期）
```

| 状态 | 含义 | 关键操作 |
|------|------|----------|
| draft | 配置/脚本未提交 | 编辑、删除、提交 |
| generating | 生成节点运行中 | 查看进度、取消 |
| partial_success | 部分模块失败 | 局部重试 |
| evaluating | 评测运行中 | 查看进度 |
| failed | 节点失败未恢复 | 重试 |
| needs_fix | 质量门禁未通过 | 查看Bad Case、修复 |
| outdated | 素材基于旧脚本 | 重新生成 |
| pending_review | 自动评测通过 | 审核、退回 |
| approved | 教师确认可交付 | 导出 |
| exported | 素材包导出成功 | 下载、再次导出 |

> **关键原则**：生成成功 ≠ 质量通过 ≠ 教师审核通过，三种状态必须分离。

## 八、评测流程

### 评测三层体系

```
层级1: 规则评测（确定性约束）
  → 字段完整性、数量、词汇覆盖、文件可播放性、版本一致性

层级2: 模型评测（语义质量）
  → 语法正确性、难度适配、连贯性、场景完成度
  → 图片场景符合度、空间关系
  → 题目可答性、干扰项有效性
  → 音频自然度与角色区分

层级3: 跨模态一致性（组合质量）
  → 文本-音频一致性（ASR回转比对）
  → 文本-图片实体/关系一致性
  → 音频-题目/答案一致性
  → 版本一致性
```

### V0.1总分公式

```
总分 = 文本质量×20% + 音频质量×20% + 图片质量×15% + 题目质量×20% + 跨模态一致性×25%
```

### 质量门禁

| 条件 | 要求 |
|------|------|
| 综合得分 | ≥80（基线测试后校准） |
| 严重错误 | 无S3/S4 |
| 素材完整 | 5项素材全部存在 |
| 答案正确 | 100% |
| 版本一致 | 全部基于当前脚本版本 |
| 安全合规 | 无学生隐私/违法/不适龄内容 |

### 一票否决项

出现以下任一情况直接判定不通过（无论总分）：
- 答案与音频相反
- 关键空间关系冲突（如方位相反）
- 音频不可播放
- 版本错配
- S4安全风险

## 九、Bad Case闭环

```
评测发现 → Bad Case分类 → 生成解释（P-BC-EXPLAIN）
→ 局部重生成/人工修改 → 回归评测 → 通过/迭代
→ 线上Bad Case每周归因 → 回流回归集 → 每月全量回归
```

### 严重程度分级

| 级别 | 含义 | 处理 |
|------|------|------|
| S0 | 无问题 | — |
| S1 | 轻微 | 无需处理 |
| S2 | 需修改 | 建议修复 |
| S3 | 不可直接使用 | 阻止导出，必须修复 |
| S4 | 安全/合规高风险 | 一票否决，阻止导出 |

### 错误类型覆盖

语法错误、超纲、目标词缺失、漏读、错读、方向冲突、题音冲突、答案错误、版本冲突

### 评测集体系

| 评测集 | 用途 | 规则 |
|--------|------|------|
| 开发集 | 调整Prompt/规则/阈值 | 可反复使用 |
| 测试集 | 冻结验证 | 禁止参与调优 |
| 回归集 | 典型Bad Case | 每次变更必须执行 |

- 建议总样本量：80条（30正常+15复杂+10边界+5信息不全+15人工注入错误+5系统异常）

## 十、Prompt管理

### Prompt清单（7个）

| 编号 | 用途 | 输出 | 版本 |
|------|------|------|------|
| P-SCRIPT | 生成双角色脚本 | JSON：角色/轮次/文本 | v1.0 |
| P-SCRIPT-EVAL | 脚本语义评测 | JSON：分数/问题/证据 | v1.0 |
| P-IMAGE | 生成情境图 | 图片+元数据 | v1.0 |
| P-IMAGE-EVAL | 图文和空间关系评测 | JSON：实体/关系/证据 | v1.0 |
| P-QUESTION | 生成听力题与答案 | JSON题目集合 | v1.0 |
| P-QUESTION-EVAL | 可答性与答案评测 | JSON：结论/证据 | v1.0 |
| P-BC-EXPLAIN | 生成Bad Case解释 | JSON：错误/原因/建议 | v1.0 |

### 设计原则

1. 明确角色：七年级英语听说课助手
2. 明确边界：不生成学生信息，不替代教师判断
3. 明确输出：强制JSON Schema，字段缺失视为失败
4. 明确证据：引用具体句子/题号/时间点
5. 明确不确定性：证据不足时输出 `manual_review`

## 十一、非功能需求

### 性能
- 端到端成功率 ≥95%
- 单节点失败自动重试1次
- 页面刷新后任务状态不丢失
- 提交按钮防重复，后端幂等

### 安全
- 不要求真实学生姓名/联系方式/成绩/录音
- 检测敏感信息时提示脱敏
- AI生成、自动评测、教师审核三状态分离
- 无教师审核不能导出
- 保留完整审计追踪（模型/Prompt/版本/操作人/时间）
- 导出页包含免责声明

### 可靠性
- 单节点失败不清空已成功结果
- 支持断点继续和状态恢复
- 局部重生成不影响其他模块

## 十二、验收要求（MVP通过条件）

1. **功能**：全部10项P0功能验收通过（AC-F01 ~ AC-F10）
2. **质量**：
   - 必选词汇覆盖率 100%
   - 素材完整率 100%
   - 标准答案正确率 100%
   - 关键跨模态冲突 0个
3. **评测器**：Bad Case Precision ≥80%, Recall ≥85%, S3/S4召回 ≥95%
4. **性能**：端到端成功率 ≥95%，单节点失败自动恢复
5. **安全**：教师控制全流程，无审核不导出
6. **闭环**：用户可稳定完成"配置→生成→评测→修复→审核→导出"
